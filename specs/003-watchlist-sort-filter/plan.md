# Implementation Plan: Watchlist Sort & Filter

**Branch**: `003-watchlist-sort-filter` | **Date**: 2026-05-05 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/003-watchlist-sort-filter/spec.md`

## Summary

Extend the watchlist page so users can sort by runtime (new), and filter by type, genre, runtime, and year range ‚Äî with all state encoded in URL query params for shareability. No schema migration required. The server handles sort ordering via Prisma; client-side `useMemo` handles all filtering. Existing type/genre filters are migrated from React state to URL params as part of this work.

## Technical Context

**Language/Version**: TypeScript 5 / Node 20  
**Primary Dependencies**: Next.js 15 App Router, Prisma 6, Tailwind v4, Radix UI  
**Storage**: PostgreSQL (no schema changes needed)  
**Testing**: Vitest (unit), Playwright (E2E)  
**Target Platform**: Web (desktop + mobile responsive)  
**Project Type**: Web application  
**Performance Goals**: Filter interactions feel instant (<100ms perceived); page load unaffected  
**Constraints**: All filter state survives page refresh (URL params); no pagination  
**Scale/Scope**: Single watchlist page; typical user has 50‚Äì500 watchlist items

## Commit Plan

### Commit 1 ‚Äî `feat(watchlist): add runtime to watchlist query and sort options`

Files: `src/app/(app)/watchlist/page.tsx`, `src/app/(app)/watchlist/watchlist-client.tsx`

### Commit 2 ‚Äî `feat(watchlist): URL-persist filter state and add runtime + year range filters`

Files: `src/app/(app)/watchlist/watchlist-client.tsx`

Changes:

- Remove `useState` for `typeFilter` and `selectedGenres`; read both from `useSearchParams()` instead
- Add `maxRuntime` and `yearFrom`/`yearTo` derived from `useSearchParams()`
- Update all filter-change handlers to push updated URL params via `router.push()`
- Update `hasActiveFilters` and `clearFilters` to cover all four filter dimensions
- Add Runtime filter control: `Select` with preset options (‚â§60, ‚â§90, ‚â§120, ‚â§150, ‚â§180 min)
- Add Year Range filter controls: two `<Input type="number">` fields (From / To) with 4-char width
- Update `filteredItems` `useMemo` to apply runtime and year range filtering
- Update empty-filter-results message to include "Clear filters" button

### Commit 3 ‚Äî `test(watchlist): add E2E tests for sort and filter`

Files: `e2e/watchlist.spec.ts`

Scenarios:

- Default sort (added_desc) renders the watchlist
- Changing sort to Title (A‚ÄìZ) reorders and persists in URL
- Genre filter shows only matching titles; URL reflects selection
- Type filter (Movies) shows only movies
- Runtime filter hides items exceeding the threshold
- Year range filter shows only titles in range
- Clear filters restores full list
- Refreshing page with active filters preserves the filtered view

## Risk Notes

- `runtime` is nullable on `MediaItem`; items without runtime are excluded when maxRuntime filter is active (documented in spec Assumptions). No special UI treatment needed.
- `year` is also nullable; same exclusion behavior for year range filter.
- Migrating `typeFilter`/`selectedGenres` from React state to URL params means users who had these filters active will lose them on this deploy (acceptable for a non-destructive read-only state change).
- The `Suspense` boundary wrapping `WatchlistClient` (for `useSearchParams`) must remain in place ‚Äî Next.js requires it. Confirm `loading.tsx` already exists (it does: `watchlist/loading.tsx`).
